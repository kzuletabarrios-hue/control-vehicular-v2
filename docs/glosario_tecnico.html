Glosario Técnico

Conceptos clave para entender cómo funciona el Sistema de Control de Acceso y Operaciones CEDI R10

Esta guía explica, en lenguaje sencillo, los términos técnicos que componen la aplicación. No es necesario saber programar para entenderlos: cada concepto incluye una explicación corta y un ejemplo dentro del propio sistema.

1. Estructura general de la app

🖥️

Frontend

Es la parte de la app que el usuario ve y toca: pantallas, botones, formularios, menús.

Ejemplo: la pantalla de "Flota Propia" donde el guarda llena los datos del vehículo.
⚙️

Backend

Es el "cerebro" que corre en un servidor: recibe lo que el usuario hace en el frontend, valida permisos, guarda y devuelve información.

Ejemplo: cuando se guarda un registro de vehículo, el backend lo valida y lo escribe en la base de datos.
🔌

API (Application Programming Interface)

Es el "menú de pedidos" que ofrece el backend. El frontend pide o envía datos usando estas "direcciones" llamadas endpoints.

Ejemplo: /api/flota sirve para crear o consultar registros de Flota Propia.
📍

Endpoint

Es cada "dirección" específica dentro de la API, dedicada a una función concreta.

Ejemplo: /api/auth/login para iniciar sesión, /api/dashboard/resumen para los datos del panel principal.
🗄️

Base de datos

Es donde se guarda permanentemente toda la información: vehículos, conductores, usuarios, registros de entrada y salida. Sin ella, todo se perdería al cerrar la app.

Ejemplo: la tabla "distribución" guarda el código, nombre y dirección de cada tienda.

2. Infraestructura — dónde "vive" la app

🟢

Supabase

Servicio en la nube que ofrece la base de datos (PostgreSQL) ya lista para usar, sin tener que instalar ni mantener un servidor propio.

Ahí se guardan todas las tablas: conductores, vehículos, usuarios, tiendas, registros, etc.
🐘

PostgreSQL

Es el motor de base de datos que usa Supabase: organiza la información en tablas con filas y columnas, como una hoja de cálculo muy potente.

🚂

Railway

Servicio en la nube donde "corre" el backend (el código Python). Cada vez que se sube un cambio al código, Railway lo despliega automáticamente.

▲

Vercel

Servicio en la nube donde "corre" el frontend (la página web). También se actualiza solo cada vez que hay un cambio en el código.

🚀

Deploy (Despliegue)

Proceso de "publicar" una nueva versión del código para que quede disponible en internet, sin que el usuario tenga que instalar nada.

🌐

Dominio / URL

La dirección web pública donde se accede a la app desde cualquier navegador.

Ejemplo: https://control-vehicular-v2.vercel.app
🔗

Proxy

Un "puente" configurado en Vercel que redirige las peticiones que empiezan con /api/ hacia el backend en Railway, para que todo funcione como un solo sitio.

3. Seguridad y acceso

🔐

Autenticación

El proceso de comprobar "¿quién eres?", normalmente con correo y contraseña.

🎫

JWT (Token de acceso)

Es como una "pulsera digital" que el sistema entrega al usuario al iniciar sesión. El frontend la guarda y la presenta en cada acción para demostrar que ya inició sesión, sin pedir la contraseña otra vez.

🛂

Roles y permisos

Una vez el sistema sabe quién eres, decide qué puedes ver y hacer.

Ejemplo: un "guarda_bodega" solo ve Flota Propia; un "admin" ve y administra todo.
🔑

Variables de entorno

Datos sensibles o de configuración (como contraseñas de la base de datos) que no se escriben dentro del código, sino que se configuran por fuera, en Railway y Vercel. Así el código puede ser público sin exponer las claves reales.

4. Código y desarrollo

📂

Repositorio (GitHub)

La "carpeta maestra" en internet donde vive todo el código del proyecto, con el historial completo de cambios.

📝

Commit / Push

Commit: guardar un "punto de control" con los cambios realizados, junto a un mensaje que explica qué se hizo. Push: enviar esos cambios al repositorio en GitHub, lo cual activa automáticamente el redeploy en Railway y Vercel.

🔄

CRUD

Las 4 operaciones básicas de cualquier sistema con datos: Crear, Reer (leer), Update (actualizar) y Delete (eliminar).

Ejemplo: crear un registro de vehículo, listar los existentes, editarlo o eliminarlo.
⚛️

React

La herramienta con la que está construido el frontend. Permite que la pantalla se actualice al instante cuando cambian los datos, sin recargar la página.

🐍

FastAPI

La herramienta con la que está construido el backend, en lenguaje Python. Permite definir los endpoints de la API de forma rápida, ordenada y segura.

5. Cómo se conecta todo

Recorrido de una acción dentro de la app

↓
🖥️ Frontend — VercelLo que el usuario ve y usa (React)
↓ pide datos vía /api/...
⚙️ Backend — RailwayValida el login, los permisos y la lógica (FastAPI)
↓ lee y escribe datos
🗄️ Base de datos — SupabaseGuarda todo de forma permanente (PostgreSQL)
